Confirmed bugs — most likely root causes

BUG 1 · HIGH
Missing env vars → Supabase client throws on init
lib/supabase/server.ts + lib/supabase/client.ts
Both createClient() calls use the non-null assertion (!) on NEXT_PUBLIC_SUPABASE_URL and NEXT_PUBLIC_SUPABASE_ANON_KEY. If either env var is missing or empty in your .env.local, the Supabase SDK constructor throws synchronously inside the Route Handler, which gets caught by the outer try/catch in GET /api/zones and returned as a 500 with "Internal server error". The fetcher in use-zones.ts then sees json.error set, throws, and SWR surfaces it as the alert you're seeing.
How to confirm
echo $NEXT_PUBLIC_SUPABASE_URL echo $NEXT_PUBLIC_SUPABASE_ANON_KEY # Both must be non-empty. Also check .env.local exists at the project root.
Fix
# .env.local (at project root, never commit this) NEXT_PUBLIC_SUPABASE_URL=https://<your-project>.supabase.co NEXT_PUBLIC_SUPABASE_ANON_KEY=<your-anon-key>
BUG 2 · HIGH
RLS blocks the anon key from reading zones
scripts/001_create_tables.sql → zones_public_read policy
RLS is enabled on public.zones and the read policy is USING (true) — which looks correct. But if the migration was run partially, or if Supabase realtime publication was added separately, the policy may not be applied yet. More critically: the server client uses the anon key (not service role), so if the schema was modified after initial setup, a missing or dropped policy silently returns 0 rows — which the route handler returns as a valid empty array, not an error. However if the table itself doesn't exist yet (migration not run), Supabase returns a 42P01 PostgreSQL error, which becomes the 500 that triggers your alert.
How to confirm
-- Run in Supabase SQL Editor SELECT * FROM public.zones LIMIT 1; -- If error: relation "public.zones" does not exist → migration not run -- If 0 rows: check seed data and RLS policies
Fix — run migration if not yet applied
-- In Supabase SQL Editor, run scripts/001_create_tables.sql -- then run scripts/002_seed_data.sql

Warnings — won't break fetch, but will cause bugs

WARNING 1
Realtime subscription uses stale data closure
hooks/use-zones.ts — useEffect dep array
The realtime useEffect lists data in its dependency array. This means the channel is torn down and re-subscribed every time SWR delivers new data — causing a subscribe/unsubscribe loop on every poll cycle. In high-traffic conditions this can exhaust Supabase realtime channel limits and cause the subscription to silently fail, which looks like stale data.
Fix — remove data from deps, use a ref instead
const dataRef = useRef(data); useEffect(() => { dataRef.current = data; }, [data]); // Inside the channel callback, replace `data` with `dataRef.current` // Remove `data` from the useEffect dependency array
WARNING 2
refreshStats() is called but never resolves to a value
app/page.tsx — handleRefresh callback
Promise.all([refreshZones(), refreshStats()]) — refreshZones() calls mutate() which returns a Promise, but refreshStats() in use-stats.ts likely also calls mutate(). If useStats doesn't expose the mutate return value, refreshStats() returns undefined, so Promise.all resolves immediately regardless of whether stats actually refreshed. Not the cause of the zones error, but the refresh button may feel broken.

Design note

NOTE
Error message wording reveals internal state
app/page.tsx — Alert component
The alert renders the raw zonesError string directly from the SWR error. If that string is ever a raw Supabase or Postgres message, it leaks schema details to the UI. The current message "Failed to fetch zones. Using cached data if available." is fine — just ensure the error passed in is always a friendly string, not error.message from an unhandled Postgres exception.

Recommended first steps

1. Verify .env.local has both Supabase vars and restart pnpm dev
2. Open Supabase SQL Editor → confirm public.zones table exists and has rows
3. Hit /api/zones directly in browser → check exact error body in response JSON
4. If fetch succeeds but realtime feels stale, apply the dataRef fix in use-zones.ts
